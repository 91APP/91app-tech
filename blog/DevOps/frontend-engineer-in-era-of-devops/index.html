DevOps 時代下的前端工程師

Ted Liang2020/06/24

今天要分享的是 DevOps 為我們團隊帶來的改變，在整個故事開始之前，我需要先介紹一下多年前我們團隊的樣貌，讓大家能夠在看見全貌的情況下，了解我們為何會用這樣的方式改變，以及我們做了什麼樣的改變

多年前

APP 製作平台

當有客戶進來使用我們的系統時，他們需要先使用 APP 製作平台，將一些 APP 相關的設定以及素材上傳，接著我們會利用自動化工具，協助他們上架 Google Play or Apple Store，如下圖：

其中，值得一提的是這部分是由兩個團隊協作完成， APP 製作團隊以及 APP 開發團隊， APP 製作團隊負責 APP 製作平台的開發，以及部分自動化工具的 api 撰寫，App 開發團隊則負責 APP 上架的功能開發以及部分自動化工具的 API 串接，也就是說假設今天要完成一個 APP 上架的功能優化，其實需要兩個團隊的協作才有機會完成。

團隊樣貌

我們有多個業務團隊，每個業務團隊負責的範圍都不盡相同，舉個例子來說 A 團隊負責的是優惠券的功能，B 團隊負責的是 UI 功能優化，C 團隊負責的是購物車相關的功能優化

每個團隊都有自己負責的項目，也因此

  • 各團隊專案開發時程不同
  • 跨團隊協作困難
  • APP 的業務邏輯越來越複雜
    • 由於各團隊業務邏輯越來越複雜，App 的 scope 也越來越大，所以越來越難讓一個人能夠去了解 APP 所有的項目
  • 部署時間長
    • 我們有超過 1000+ 以上的 App，假設每個 APP build 10 分鐘，10 * 1000 也是要一段時間
  • 測試團隊重複測試
    • 這其實牽涉到我們的 git flow 問題

Git Flow

我們過去是使用 git flow，也就是會有 feature branch / develop / master brabch ，然後每個團隊都會擁有自己的 feature branch，如下圖：

舉個例子來說 A 團隊會使用最上面橘色的 branch，B 團隊會使用綠色的 branch ，C 團隊會使用藍色的 branch，直到要 release 前大家才會把所有的程式合併回 develop branch。

這也造成了，測試人員為了避免在 release 前才發現問題，所以必須在各團隊的 branch 上提早做測試，然而真正要 release 出去的版本其實是 develop branch 的東西，也因為這個原因，測試人員就必須在團隊的 branch 上測試一次，大家合併完，解完衝突後還要再測試一次。

回到現在，我們可以怎麼解決我們的問題?

參考一些書籍，DevOps 實踐指南 這本書裡面提到：

建立從產品計畫直至功能上線的端到端的快速服務交付流水線

所以 DevOps 就是 CI/CD ? 不是的！

DevOps 其實是：

  1. 建立文化
  2. 賦能給團隊
  3. 讓團隊有能力改善現場的問題

賦予團隊彈性，讓團隊有能力改善現場遇到問題的一種方法。

DevOps 三步工作法

DevOps 三步工作法，分別是：

  1. 流動原則：設法看見事情的全貌，看見所有的工作項目，看見所有的浪費，看見所有的價值流，並且設法去消除浪費，消除等待，消除不必要的事情，然後持續的去改善
  2. 反饋原則：就像是敏捷的概念，小包迭代，避免花了兩三個月做完一個成品後，卻發現客戶要的不是這個東西，如果可以的話，盡可能在兩週內生出一個 prototype，跟客戶確認這個東西是他要的，然後才持續的改進 / 優化；再來是確保產品的品質，盡可能在一開始的時候就做好品質的確保，避免不良品流向下游，舉個例子來說，假設有個 bug 在交付到使用者手上才被發現，那這樣造成的影響是從開發/測試/業務/維運人員都需要花費時間在處理這個問題上，如果可以在一開始開發完成的時候就做好品質的確保，這樣就可以避免大家浪費時間成本在這件事情上。
  3. 持續學習與實驗原則：建立持續學習/持續改善/持續分享/安全的文化，值得一提的是，越安全的文化，團隊越敢說實話，團隊越敢說實話就越有機會能夠發現問題，並且改善他。

回到我們的問題

APP 製作與 APP 開發團隊的問題是什麼?

  • APP 製作平台需要了解 APP 開發團隊的 domain know how
    • APP 製作團隊需要了解 andorid/ios 怎麼上架/怎麼 Build APP ，需要了解各平台的 domain know how，然而這些自動化流程又不是他們寫的，他們卻需要了解那麼多負責的內容
  • APP 開發團隊想幫忙卻幫不上忙，App 製作團隊使用的開發語言並不是 APP 開發團隊熟悉的語言，即使 APP 開發團隊想協助撰寫 API，也因為語言不通而幫不上忙
  • APP 開發團隊沒有能力優化自己的流水線，假設今天 APP 開發團隊想優化自己的流水線，就會需要跟 APP 製作團隊協調資源，如果 APP 製作團隊沒有資源就無法優化自己的流水線

所以問題是什麼? 問題是 APP 開發團隊沒有能力 end to end 的解決自己的問題。

所以我們做了調整

我們讓 APP 製作團隊跟 APP 開發團隊的職責拆分

  1. APP 製作團隊只負責客戶的素材設定部分
  2. APP 開發團隊負責 build / 上架 / 流水線的功能開發

舉個例子來說：假設今天一個客戶需要製作一個 APP，他們會先利用 APP 製作平台設定他們的素材，接著當他需要做建置 APP 或上架 APP 的時候，APP 製作平台會發一個 event 給 APP 開發團隊的 Build Service，或上架 APP Service，讓那些 Services 負責去處理相關的事情

利用這樣的職責分離，我們讓 APP 製作團隊的人再也不需要去了解複雜的 domain know how，而 APP 開發團隊能夠自己掌握自己的 build/上架/流水線流程

藉由這樣的方式，我們讓團隊真正達到了 賦能 的效果，讓團隊能夠 end to end 的處理自己的事情

測試團隊要測試多次的問題

回到我們的 git flow 這種 git flow 的優點是什麼？缺點是什麼？

  • 優點
    • 開發流程簡單：大家只需要專注在自己的 branch 上，不會擔心被別人影響
  • 缺點
    • 測試人員要測很多次：因為大家都會把 code 放在外面大約一兩週以上，所以測試人員為了避免合併進 develop 才測試，所以會提早在團隊 branch 上做測試，然而在團隊 branch 上做的測試其實是沒有意義的，因為真正要 release 出去的版本其實是合併後的那包，測試團隊的人員卻需要花非常多的時間在測試將來可能還是有問題的東西上
    • 衝突很大包：每次大家進 code 到 develop 的時候，很容易造成 conflict，而且這個時候通常距離 release 時間很近，所以開發人員和測試人員的壓力都會很大，開發人員需要在很短的時間內解決 code conflict，然後測試人員需要在更短的時間內驗證是否有因為解衝突造成額外的問題，另外更難發現的是，可能 A/B/C 團隊的東西在合併之後才發生的問題，也因為測試時間過短而沒有被發現

所以問題是什麼?

  • branch 待在外面太久：各種書上都有提到應該盡可能提早將程式合併進 develop，我們有沒有機會改善這件事情?
  • 測試團隊測試多次：我們有沒有機會避免測試多次?
  • 所有的團隊改同一份 codebase：由於各團隊的業務目標都不同，但是因為都是改在同一份 codebase 上，所以很容易造成 conflict，我們有沒有機會將程式碼分離?

調整 Git Flow

根據最近 Martin Fowler 的文章有提到，大概有分成兩種分支策略

  1. Feature Branching
  2. Continuous Integration

第一種比較接近我們團隊分支的方式，等到功能開發完成才合併 (merge) 進 develop，這種方式的特色是通常合併頻率相對較低，造成的衝突也會比較大，也就是下圖的黃色的點。

第二種的話則是如果有個 commit 可以正常執行且不會影響到別人，那就可以進到 develop branch，這樣的特色是合併的頻率高，造成的衝突通常較小。

所以我們做了調整。

導入 Trunk Based

我們導入了 Trunk Based Development (TBD)，他其實是 Continuous Integration 的一種方式，盡可能的提早將程式碼併進 develop。然後 TBD 有一個特色是，在 release 前會開立 release branch，在 release branch 上會用盡各種方式確保這次只有要 release 出去的東西才會被發布出去。

舉個例子來說：假設有個 A 功能在 release branch 開立前就已經進到 develop，可是這個功能並沒有要 release 到下個版本，這個時候就有幾種方式

  1. git revert
  2. feature toggle
  3. 各種骯髒手段

如果今天的 code 相對單純那就可以使用簡單的 git revert 去把合併 revert 掉，但假設今天這個 A 的功能是由多個合併完成的，那可能就無法簡單地用 git revert 去 revert 掉，那這個時候就必須使用 Feature Toggle 去動態的將功能做開關。

feature toggle: 簡單的說就是 if-else 邏輯，如下：

if(flag_A) {
    do_A()
} else {
    do_nothing()
}

然而 feature toggle 的實作都會使用到一些技巧例如：interface / abstract / strategy pattern 等方式。

然而假設今天前面兩種方式都沒辦法簡單的將功能移除，那就只能用比較骯髒的手段去關閉他，舉個例子來說假設這個功能有個統一的入口，那我就會想辦法把那個入口拿掉，又或者這次新增的功能是：

if(xxx) {
    xxx()
} else {
    newfeature()  // new feature
}

那我就會直接發一個 Pull request (PR) 把 new feature 這段 code 給註解掉，簡單的說就是用盡各種手段，確保 release branch 上只有這次要 release 出去的功能。然而，也因為我們會在 release branch 上做各種奇怪的手段，所以 release branch 在 release 後會直接廢棄，不會合併回 develop。

所以為了確保這次修復的 bug 在下個版本也被修復，我們就會在 develop 將 bug 修復後使用 cherry-pick 的方式 cherry-pick 到 release branch，這樣就可以確保 bug 在這個版本被修復，在下個版本也被修復。

另外，對於測試人員，在 release branch 開之前，他們只需要測試 develop branch， release branch 開之後，他們只需要測試 release branch，藉由這樣的方式，就解決了測試人員需要在多個 branch 上測試的問題。

實際上的作法

剛剛講了很多是 TBD 的 Best Practice ，然而我們在實際運作的時候還是有做一些調整：

  1. 我們並沒有每個 commit 就直接發 pull request 進 develop，主因是如果每個 commit 就發 PR 進 develop，codeReview 的速度就會變得很關鍵，然而我們並沒有那麼快的 codeReview 速度，為了避免造成困擾，我們還是維持小包，而不是每個 commit 都發 PR
  2. 我們目前整個流程都還是人工介入較多，並沒有將整個流程自動化，舉個例子來說：假設今天 release branch 創建後，會由我們安排的 release manager 來提醒大家記得要做 cherry-pick，新人也會有相對應的教育訓練，讓他們知道我們的 git flow 應該怎麼走
  3. 在開發階段，我們並不會將所有功能都加上 feature toggle，主因是 feature toggle 某種程度上會增加開發的時間，為了避免浪費不必要的時間，我們會保持一些彈性，利用溝通的方式來解決這個問題，舉個例子來說：假設我有一個功能希望不要加上 feature toggle，我可能會讓團隊成員知道我目前正在修改的範圍，跟大家確認是否會有影響，如果不會影響到大家，我就會避免使用 feature toggle
  4. 我們還是會有 branch 放在外面較久，承第三點，我們會保持一些彈性，利用溝通的方式，讓大家可以盡可能好做事情，所以只要在能夠滿足幾個原則之下，我們就可以有特例
  • 讓團隊成員了解整件事情的全貌
  • 不影響到其他人

我們使用 TBD 的目的是為了：

  1. 降低 code衝突的機會
  2. 降低測試人員測試多次的問題.

只要在滿足這兩點的情況下，我們可以利用每天的 daily stand up 去同步團隊成員在做的事情， 利用這個場合讓大家討論正在開發的功能有沒有必要開 feature toggle，或者是可不可以把 branch 過一陣子再合併，會不會影響到其他人，藉由這樣的討論，我們就有機會可以保持一些彈性，避免無意義的制度化。

如何確保品質

由於我們的 code 會非常頻繁的合併進 develop branch，如果有不健康的程式合併進 develop 會影響到非常多的人，如下圖：

當程式合併進 develop，通常會直接發動 deploy qa 環境，然後其他的 RD 可能會把 code checkout 去做修改，如果有不健康的 code 進到 develop，就會影響到這些人，所以為了確保 code 的健康程度，我們就有必要做一些確保，如下圖：

當發 PR 時，我們會自動跑 unit test stage and lint stage 去確保程式碼的品質，在某些情況下，我們也會主動的跟測試人員要測試項目，先主動測試過後才發 PR，如果做了這些確保，merge 進 develop 的 code 還是有問題，那我們就會迅速的 revert and rollback 避免影響到太多的人：

詳細的步驟就會是

  1. 發 PR 後會跑 unit test and lint stage 確保品質
  2. 進 develop 後自動 deploy qa 環境
  3. 接著會跑 end to end 的 integration test 確保沒問題
  4. 如果 integration test 有發現問題就會自動 rollback 或者發出 alarm

做這些的目的都是為了降低影響到其他團隊成員的時間。

TBD 的好處

  1. 衝突範圍變小
  2. 測試只需要測試一次
  3. 部署信心度提升：每天進 develop 就會 deploy，每天 deploy 好幾次，信心度高
  4. RD 主動確保品質：為了避免影響到其他團隊成員，RD 會主動確保品質

架構

我們除了使用 TBD 來解決程式衝 的問題外，我們也研究有沒有機會使用架構來解決這些問題：

康威定律：產品的架構通常會對應到組織的架構，工作溝通方式，如果刻意違背則會增加溝通的障礙

所以我們來 recap 一次我們的團隊樣貌：

我們將原本大單體的程式碼，切分成多個模組我們得到的幾個好處

  1. 產生領域專家
  2. 限縮修改範圍
  3. 保持彈性
  4. 降低程式衝突

產生領域專家 and 限縮修改範圍

由於每個團隊都 focus 在解決他們的業務目標，所以通常各團隊調整的範圍也都會被限縮在某幾個模組內，舉個例子來說：A 團隊通常只會修改會員模組以及登入模組

利用這樣的方式我們就可以在各團隊中產生領域專家，這些人會對這些模組以及邏輯特別了解，藉由這樣的方式，我們可以有多個領域專家去了解各領域的問題，再也不需要有一個人去了解整個 APP 的功能。

保持彈性、降低衝突

由於我們只有拆分模組，並沒有拆分服務，所以各團隊的成員還是可以直接去修改其他團隊的模組，另外，當修改到的模組不是自己擅長的，還可以直接請教該模組的領域專家，藉由這樣的方式，我們可以讓團隊快速的分享知識，並且達成目標。

其他的改善項目

編譯超過一千個 APP 需要花很多時間，原本機器放在機房，編譯的機器大約 5-6 台，這樣有這些問題：

  • 成本高
  • 機器隨時可能壞掉
  • 再好的機器放 3-5 年也會變成普通的機器

藉由將機器搬到雲端，讓我們可以擁有無限的資源，原本編譯 APP 需要花 1 天以上，上雲端後，我們可以一次啟用 80 台 Spot Instance 機器，在 3 小時內就可以完成，並且成本可以控制在 10 USD/次

結論

  1. 我們建立了互相學習的環境，讓團隊可以互相學習其他團隊的技能，並且有機會運用在自己團隊的問題上
  2. 我們將測試以及維運變成自己生活的一環，藉由 TBD 我們會避免將不正常的程式合併到 develop，我們會盡可能地將測試左移，降低不良品發生的機會，另外我們也有維運自己的服務，我們也會設法去優化維運遇到的問題
  3. 藉由讓 APP 開發團隊能夠 end to end 去處理 build 流程 / 上架流程，來降低等待時間
  4. 設法讓團隊資訊同步，讓團隊成員看見全貌，來保持彈性，避免一昧的制度化
  5. 藉由以上這幾點，讓團隊成員能夠學習其他團隊的技能，了解其他團隊的痛苦，藉由讓團隊成員看見全貌，想辦法讓團隊能夠 end to end 的處理他們的事務，藉由這樣的方式，讓團隊真正 有能力 去改善他們遇到的問題

藉由賦能的方式，建立一個自組織的團隊，我認為這就是 DevOps 帶來的改變
